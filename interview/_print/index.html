请说下多态, 重载, 重写

多态:

多态指的是执行一段代码, Java可以根据运行时对象类型产生不同的结果 也就是不同对象上调用同一个方法, 有不同的结果 多态实现的前提有三个

  1. 继承关系, 要有父类
  2. 重写父类方法
  3. 父类引用指向子类的对象

父类定义一个方法, 子类可以自己提供实现, 子类对象调用父类的方法就可以重新引用到具体的子类方法, 从而实现同一方法, 不同结果的多态

子类自己提供实现 这个行为叫做重写父类方法

重载:

在同一个类中，允许定义多个方法名相同但参数不同的方法, 叫方法重载, 方法重载可以提高程序的可读性, 对于同一种逻辑, 只是传入的参数不同没必要用不同的名字, 代码更简洁 比如计算矩形面积, 传入两条边的边长就是计算矩形面积, 只传一条边长, 就是计算正方形面积 这两个方法的名字都可以是 area

重写例子:

// 父类：动物
class Animal {
    void makeSound() {
        System.out.println("动物在叫");
    }
}

// 子类：猫
class Cat extends Animal {
    void makeSound() {
        System.out.println("猫在喵喵叫");
    }
}

// 子类：狗
class Dog extends Animal {
    void makeSound() {
        System.out.println("狗在汪汪叫");
    }
}

重载例子:

public class Shape {

    // 计算矩形的面积（长和宽）
    public int area(int length, int width) {
        return length * width;
    }

    // 计算正方形的面积（只给定一条边）
    public int area(int side) {
        return side * side;
    }
}

继承和抽象的区别

继承允许子类直接复用或者重写父类方法, 通过继承子类可以重用父类代码

抽象是隐藏复杂性和只显示必要部分的技术, 通过抽象类和接口实现

hashcode 和 equals 方法只重写一个行不行，只重写 equals 没重写 hashcode，map put 的时候会发生什么

不行

两个对象通过equal相等, 则他们的hashcode必须相等

这是因为map的key是由hashcode计算而来的, 在map中两个对象相等代表着key相同, 只重写equals在mapout时会发生键值冲突或者元素重复

Java 编译时异常和运行时异常的区别

编译时异常, 在编译时必须显示处理, 这是由程序员手动抛出或手动捕获的 如果某段代码可能会抛出异常, 比如关闭线程, 那么编译器强制要求手动捕获, 否则编译不会通过

运行时异常, 这类异常在运行时抛出, 是 RuntimeException 的子类, 对于运行时异常, Java编译器不强制要求处理

比如空指针错误, 数组溢出, BeanNotDefinaion等

收录的京东同学 10 后端实习一面的原题：return 先执行还是 finally 先执行

finally 先执行, 无论try中执行什么, 都是finally先执行 也就是说一旦 finally 要求 return 那么程序将直接返回, 不会执行try-catch块里的函数

请说说 MySQL 的内联、左联、右联的区别。

内联就是返回两张表都匹配的记录, 只返回两张表都存在的记录 左联, 返回左表所有记录, 不管右表是否有相匹配的记录,如果没有结果集中右表就填NULL 右联, 返回右表所有记录, 不管左表有没有匹配的记录

订单表, 订单顾客映射表, 可以通过订单ID关联

内联

SELECT LEFT(a.title, 20) AS articleTitle, LEFT(c.content, 20) AS
CommentContent
FROM order a
INNER JOIN ORDERCONSUMER c ON a.id = c.order_id
LIMIT 2;

左联

SELECT LEFT(a.title, 20) AS ArticleTitle, LEFT(c.content, 20) AS CommentContent FROM order a LEFT JOIN ORDERCONSUMER c ON a.id = c.article_id LIMIT 2;

右联

SELECT LEFT(a.title, 20) AS ArticleTitle, LEFT(c.content, 20) AS CommentContent FROM comment c RIGHT JOIN article a ON a.id = c.article_id LIMIT 2;

    TreeMap 基于有序红黑树实现的map映射, 元素按照增序排列

@autowried和@resource的区别

来源: 第一个是spring的注解 第二个是JavaEE注解, 是Java规范的一部分, Spring提供了具体实现,

查找顺序: 前者的 默认注入方式是按照类型匹配, 首先在 IOC 容器里找接口类型匹配对象,因为接口会有多个实现类所以容器里可能会有多个相同接口类型的 Bean.

找到多个就按照名称匹配, 名称默认首字母小写, 但是这里的名称除了是实现类实例名称之外还可以用 @Qualifier 注解指定, 注解指定的名称优先级最高, 按注解名称进行匹配

/**IOC**/
A1 a1;
A2 a2;
/**IOC**/

@Autowired //无法注入
private A a;

@Autowired //无法注入, 约束名称未找到
@Qualifier(value = "a")
private A a2;

@Autowired //可以注入
@Qualifier(value = "a2")
private A a;

当然也可能没有匹配到类型, 那么就抛出 notDefination异常

后者默认按照名称匹配, 匹配到多个结果就做类型匹配, 没有结果就按类型查找

@Resource注解内部有两个属性, name, type

public @interface Resource {
    String name() default "";
    Class<?> type() default Object.class;
}

如果注解只指定名称, 就按照名称匹配, 只指定类型, 就按照类型匹配

// 无法注入 noUniqueBeanDef
@Resource
private A a;
// 正确注入
@Resource
private A a1;
// 正确注入 对象对应的 bean（比较推荐这种方式）
@Resource(name = "a1")
private A a;

Spring常用注解

web
  • @RequestMapping 请求路径映射, 符合路径的交由控制器处理 类似的有 @PutMapping,@GetMapping,@PostMapping, @DeleteMapping 四种分别对应四种请求
  • @PathVariable 获取路径里的参数 @RequestParam 获取查询的参数
  • @RequestBody 读取Request请求的body部分并且
AOP
事务
容器
  • @Autowired 自动导入对象到类中, 被注入进的类同样要被Spring容器管理, 比如service类注入到controller类中
  • 反过来, 将某个类交给到Spring容器管理也有对应的注解 @Component, @Service, @Controller, 作为声明式接口, 分别用于标记组件, 服务bean, 返回视图的控制器bean
  • @RestController 声明式注解, 标记该类是一个控制器, 但是 @RestController 可以视为 @Controller 和 @ResponceBody 的集合, 将返回值直接写入 HTTP 请求体里面
  • @Scope 声明域注解, 声明一个Bean的作用域, 有四种 singleton 单例, prototype 每次访问创建新的实例
  • @Configuration 当成 @Component 就行, 可能这个更直观点
